# Technical debt

Single source of truth for technical debt in this repo. Debt that takes
more than a few minutes to clear should land here (or in a GitHub issue
labelled `tech-debt`) so it does not get forgotten.

## How to use this file

- **Add an entry** when you ship code that you know is incomplete,
  taking a shortcut, or putting off a refactor.
- **Always write the resolution condition.** A debt entry without a
  trigger to clear it is just a complaint.
- **Mark the source.** Commit SHA or PR number, so future readers can
  see where the debt came from.
- **Move resolved items** from Open to Resolved (don't delete) so the
  history of what we paid down stays visible.
- **GitHub issue vs. `dashboard.md`**: use the `tech-debt` issue label
  for debt that needs design discussion or that spans more than one
  area. Use this file for debt that is mostly self-contained and just
  needs to be remembered.

## Scanner

[`debt-scan`](../../crates/debt-scan/) counts machine-detectable
debt categories (TODOs, `#[allow(dead_code)]`, ignored tests,
personal paths, external dependencies, …). It runs locally and in CI.

```sh
cargo run -p debt-scan -- scan      # write reports/debt/latest.{md,json}
cargo run -p debt-scan -- gate      # fail if any category went above baseline
cargo run -p debt-scan -- gate --update-baseline   # accept new baseline
```

| File                                | Role                                                         |
| ----------------------------------- | ------------------------------------------------------------ |
| [`baseline.json`](baseline.json)    | Locked-in counts. Bumping it is a deliberate choice — explain why in the commit message. |
| [`allowlist.json`](allowlist.json)  | Reserved for per-path false-positive overrides (not yet wired). |
| `reports/debt/latest.{json,md}`     | Generated by `debt-scan scan`; gitignored.                   |
