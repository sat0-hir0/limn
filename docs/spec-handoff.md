# ブロックエディタ 技術スタック & 設計仕様（Claude Code 引き継ぎ書）

> 思考を深めるための、シンプル・キーボードファースト・AI統合のブロックエディタ。
> このドキュメントは設計議論の結論をまとめた実装の出発点。

---

## 1. このプロダクトは何か（一行定義）

**フォルダ内の `.md` ファイル群を、キーボードファーストで編集する、シンプルでAI統合されたエディタ。**
Obsidian の思想（md ファイル群＝知識ベース、リンクで繋ぐ）を土台に、書く体験を独自のシンプルさで作り替えたもの。

---

## 2. コア要件（確定）

| # | 要件 | 意味 |
|---|------|------|
| 1 | **Markdown で保管** | 保存フォーマットが `.md`。独自バイナリ/DBに閉じ込めない。他ツールで開ける・Git差分が見える・ロックインなし |
| 2 | **ブロック追加は `/` コマンド** | 新規ブロックはハンドルやボタンでなく `/` から呼ぶ。マウス不要 |
| 3 | **編集もキーボードファースト** | 既存ブロックの編集・移動・変換も同じ思想。追加と編集のUIの一貫性 |
| 4 | **ナレッジ管理（A/B/C 全部）** | A:リンク（Obsidian型）/ B:階層（フォルダ）/ C:日付流し（デイリーノート）。土台は「フォルダ内の `.md` 群」一つで全部成立 |

### 「シンプルにしたい」対象（最優先の価値観）
- **UI** — 画面に余計なものを出さない。IA Writer 的な静けさ。ハンドル/ホバーメニュー/ツールバーを出さない
- **キーボードファースト** — 手をホームポジションから離さない。`/` が司令塔。マウスは使えるが主役でない。Vim のモード制は不採用（モードレスで速い＝Zed 路線）
- **AI統合** — AI が後付け機能でなく書く体験に溶け込む。ただし盛りすぎない

---

## 3. 確定した設計判断

- **背骨は線形ブロック** — 縦一列が基本。座標管理（自由配置キャンバス）は持たない
- **ブロックは「裏方の構造」** — 普段はただのテキストに見え、ブロックUIを画面に出さない。`#` `-` 等の Markdown を打つと構造化される（"打って化ける"）
- **`/` が操作の入口** — ブロック追加・種類変換・移動などを `/` から呼ぶ。覚えるのは最小限
- **表現は豊かに、到達は一瞬に** — URL/テーブル/画像もフルサポート。ただし到達手段は「打って化ける」＋「`/` で呼ぶ」。表現は削らず、操作だけ削る
- **空間は妥協** — 専用キャンバスは作らない。リンクから自動生成される **グラフビュー**＋**フォーカスモード**（一点を明るく他を薄く）で「関係を見る/集中して潜る」欲求に応える
- **保存はサイレント** — 「保存ボタン」概念なし。自動保存

### 未決（Claude Code 側で決める／実装しながら詰める）
- **AI統合の絡ませ方** — 第一候補は **「選んで頼む」**（テキスト選択→ `/` かショートカットで要約・続き生成。自分が主導、AIは勝手に出ない＝最もシンプル）。他案：カーソル直下のAI指示エリア／裏で先回り（Copilot型、シンプルさと緊張）
- **`/` の範囲** — 追加専用か、移動/変換/削除まで全部 `/` から呼ぶ万能入口にするか
- **補完の「呼吸」** — 候補をいつ出すか。シンプルさ優先なら「普段は黙る、呼ばれたら出す」に倒れる。実機で要調整

## 4. 技術スタック

作り方には2路線ある。**シンプルさ・最速で形にする**なら路線A、**学習意欲・思想を貫いて全部自作**なら路線B。

### 路線A：Obsidian プラグイン（最速・最シンプル）
要件1（md保管）と要件4（ナレッジ管理 A/B/C）は Obsidian が最初から持つ＝**作らなくてよい**。自分は要件2・3（`/` 司令塔・書き味）と AI統合だけに集中できる。

| 領域 | 採用 |
|------|------|
| 土台 | Obsidian（md・リンク・グラフ・モバイル同期を流用） |
| 言語 | TypeScript |
| 形態 | Obsidian Plugin API |
| md・リンク・グラフ | **Obsidian 標準（自作しない）** |
| AI | 各社 LLM API を非同期呼び出し |

- 利点：最速。md/リンク/グラフ/同期がタダ。GUIエンジンを建てる必要なし（テキスト描画・IME・無遅延描画ループの苦労が消える）
- 妥協：Obsidian（Electron）の作法と性能天井に縛られる。「Zed級の軽さ」は出ない

### 路線B：Rust ネイティブ（思想を貫く・学習向き）
「Zed みたいに軽量・無遅延」を妥協しないならこちら。代償は開発量が大きい。

| 領域 | 採用 | 備考 |
|------|------|------|
| 言語 | Rust | |
| GUI | gpui-component ベース＋本家 gpui で補完 | gpui は pre-1.0・破壊的変更頻繁。gpui-component に LSP対応エディタ/Markdownレンダリングのサンプルあり、これを出発点に |
| ドキュメントモデル | 自作 | ブロックツリー（線形＋カラム＋畳み）。テキストは Rope 構造 |
| 永続化 | `.md` ファイル入出力 | 保存ボタンなし・自動保存 |
| AI | 非同期（裏スレッド） | 主スレッドを絶対に待たせない |

- 注意：UI層を薄く保ち gpui 依存を局所化しておく（破壊的変更への保険）
- 不採用：egui（即時モードでリッチテキスト編集に不向き）／wgpu からの完全フルスクラッチ（テキストレンダリング・IME・再描画最適化で年単位の遠回り。学習トラックとして別途温存）

### アーキテクチャ原則（路線B採用時、3層）
```
UI層（gpui）          : 書く地点／補完候補／AI指示エリア  ← 主スレッド・無遅延を死守
コア層（自作）         : ブロックツリー／編集オペ／補完エンジン
サービス層（裏スレッド）: 永続化(.md)／AI呼び出し／索引・バックリンク  ← 遅くてよい
```
**境界線：主スレッドは絶対に待たない。** AIも保存も裏に逃がし、結果は非同期で主へ返す。
速い補完（コア層・同期・ルールベース）と賢いAI（サービス層・非同期）をスレッドで物理的に住み分ける。

## 5. 補完エンジンの設計（路線共通の考え方）

肝：**「候補を作る人」と「いつ出すか決める人」を分離する。** 呼吸の調整は後者だけ差し替える。

```
入力イベント → コンテキスト抽出 → 候補プロバイダ群（複数の小さな専門家）
   → ゲート（集約・順位付け・"今は黙る"判断）
   → ポリシー（呼吸：いつ/何個/どれだけ待つ ← ここだけ差し替えて調整）
   → 提示（カーソル直下に候補）→ 受領(Tab)/棄却(Esc)
   → 学習ログ（採否を記録しポリシーに反映。最初は入れ物だけ）
```
- プロバイダは複数並べる（`#`→見出し、リスト途中→次項目、URL検知→リンクカード…）。後でAIプロバイダを1個足せる構造に
- ルールベースなら全部同期・無遅延。AIプロバイダは非同期で後から候補を差し込む
- シンプルさ優先の初期方針：**普段は黙る。`/` や `#` で呼ばれた時だけ出す**

---

## 6. Claude Code への最初の依頼例

путь路線が決まっていれば、こう頼むと早い：

- 路線A：「Obsidian プラグインの雛形を作って。`/` コマンドパレットから自作ブロックを挿入できる最小構成から」
- 路線B：「gpui-component のエディタサンプルを clone して動かし、`.md` ファイルを読み込んで表示する最小構成を作って」

最初のマイルストーン（共通）：
1. `.md` を1枚開いて表示できる
2. キーボードで編集して保存できる（自動保存）
3. `/` を押すと候補が出てブロックを挿入できる
4. テキスト選択 →ショートカットで AI（要約/続き生成）を呼べる ← AI統合の第一歩

## 付記：このプロジェクトの背骨（議論の結論）

最初は「キャンバス型に惹かれるが操作感が死ぬ」という矛盾から出発した。
辿り着いた答えは、**自由配置の空間を諦め、線形ブロック＋リンク（グラフビュー）＋フォーカスモードで"空間の旨味"だけを取る**こと。
そして全要件は **「フォルダ内の `.md` ファイル群を、キーボードファーストで編集する」** の一文に集約される。
シンプルさ（UI・キーボードファースト・AI統合）が、全設計判断のモノサシ。
