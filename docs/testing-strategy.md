# テスト戦略（推奨まとめ）

> ブロックエディタのテスト戦略。アーキテクチャ（Functional Core, Imperative Shell ＋ ドメイン整理）と接続した形でまとめる。
> 結論：トロフィーをベースに、純粋コアの単体は手厚く（ハイブリッド）。進め方はコアTDD気味・探索部はテストアフター。

---

## 1. 全体方針

- **型**：テストトロフィーをベースにする（統合テストを主役・最も厚く）。ただし `Functional Core` の純粋ロジックは単体テストを手厚く持つ＝ピラミッドとのハイブリッド。
- **進め方**：仕様が固い所はTDD気味（先にテスト）、仕様が「やってみないと分からない」所はテストアフター（作ってからテスト）。
- **Rust前提**：静的解析（型・clippy・所有権チェック）が分厚く、他言語ならテストで捕まえるバグの相当部分をコンパイル時に潰す。トロフィー最下層が書かずして厚い。

理由：このエディタは二つの性格を持つ。`Functional Core`（ブロック操作・md変換・補完ルール）は純粋ロジックの塊でピラミッド的＝単体が効く。UI・ファイル・AIの結合部はトロフィー的＝繋いで初めて意味が出る。だから両者を使い分ける。

---

## 2. 4層の中身

### 静的解析（最下層・無料の番人）
- 型チェック / `cargo clippy` / 所有権チェッカー
- null参照・解放済みメモリ参照が原理的に起きない
- 書かずして分厚い。CIで `clippy` を必須に

### 単体テスト（土台・大量・手厚く）
- 対象：`core/` 配下の純粋関数
- 例：
  - `block`：挿入・移動・畳みオペが期待ツリーを返すか
  - `markdown`：**md → ツリー → md のラウンドトリップで情報が落ちないか**（最重要・最優先）
  - `completion`：`#` 入力で見出し候補が出るか（ルールベース部）
  - `link`：`[[ ]]` の解析が正しいか
- 純粋関数なので入出力だけ見ればよい。セットアップ不要・爆速。Rust標準の `#[test]` を各モジュール隣に

### 統合テスト（主役・最も厚く）
- 対象：`core` ＋ `shell` を結合したユーザーシナリオ単位
- 例：
  - ファイルを開く → ブロック挿入 → 移動 → 自動保存されたmdが正しい
  - `[[link]]` を書く → バックリンク索引が更新される
  - `/` コマンドでテーブル挿入 → mdに正しく反映
- 実装の細部でなく振る舞いを検証＝リファクタに強い

### E2E（頂点・少数・生命線のみ）
- 対象：gpui を起動し実キー入力をシミュレート、画面状態を検証
- 例：起動 → `# ` と打つ → 見出しとして表示される／編集して保存できる
- 数を絞る（遅く・脆い）。クリティカルな数本だけ
- **注意**：gpui はネイティブGUIで Web の Playwright 等が無い。gpui 自身のテスト機構（`TestAppContext` などヘッドレスでイベントを流す仕組み）を使う。ここは着手前に要調査

---

## 3. 進め方（いつ書くか）

| 対象 | 進め方 | 理由 |
|------|--------|------|
| `markdown` 変換・`block` オペ | **TDD気味**（先にテスト） | 仕様が固い。入出力が明確 |
| `link`・`vault` | TDD気味 | 仕様が比較的明確 |
| 補完の「呼吸」・UI挙動 | **テストアフター** | 「やってみないと分からない」探索的領域。仕様が動く |
| 統合シナリオ | BDD的に（「〜したとき〜になる」） | トロフィー統合層と相性良い |

---

## 4. このプロジェクト固有の最重要テスト

- **ラウンドトリップテスト（md ⇄ ツリー）が生命線**。「内部はブロック、保存は `.md`」という設計上、変換で情報が落ちないことが死活的。単体テストで網羅的に。多様な記法（見出し・リスト・コード・テーブル・リンク・画像・ネスト）を入力に
- **Undo/Redo の整合性**。編集オペ適用 → undo で元に戻るか。ブロック移動の undo まで
- **補完ルールの回帰**。ルールを足すたびに既存候補が壊れないか

---

## 5. CI で回す順序（速い順）

1. `cargo clippy`（静的解析）— 一瞬
2. 単体テスト（純粋コア）— 速い
3. 統合テスト（core＋shell）— やや遅い
4. E2E（gpui起動）— 遅い・最後

下ほど頻繁・上ほど絞る。ローカルでは1〜2を常時、3〜4はpush時。

## まとめ（一行）

**トロフィーをベースに純粋コアの単体を手厚く。ラウンドトリップテストを生命線に据え、固い所はTDD・探索的な所はテストアフター。Rustの静的解析を最下層の番人として活かす。**
